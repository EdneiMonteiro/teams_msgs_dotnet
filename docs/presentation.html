Motivação

Por que Bot Framework — e não Power Automate ou Graph?

Comunicação corporativa em massa via Teams exige o canal certo. A demo não burla rate limits: usa proactive messaging do Bot Framework.

AbordagemRealidade
Power AutomateLimites por licença e conector; não projetado para processamento massivo — sofre throttling e latência.
Microsoft GraphThrottling multi-nível (app, tenant, user); adequado para integração, não para broadcast massivo.
Bot Framework ✅Canal nativo para notificações e proactive messaging; melhor suporte a escala, com rate limits dinâmicos.
O envio depende do Teams App estar instalado por usuário (org-wide via Admin Center). O bot captura uma conversationReference por usuário e a usa depois para mandar mensagens 1:1 sem nova interação.
Big picture

Dois serviços, Service Bus no meio, Redis no caminho quente

teams-msgs-api

  • ASP.NET Core Minimal API · HPA 1..5 por CPU
  • Recebe webhooks do Bot (/api/messages) e grava refs
  • POST /api/send: streaming das refs + batches no Service Bus

teams-msgs-worker

  • Worker Service (BackgroundService) · KEDA 0..10
  • Consome Service Bus em PeekLock; dedup nativa via messageId
  • Token bucket Redis antes do ContinueConversation → Bot Framework (1:1)

Fila

Service Bus send-messages: batches, dedup nativa + DLQ

Estado

Table refs (conversationrefs) + Redis: counters, índice, cache, bucket

Ingress

Istio Gateway + cert-manager + Let's Encrypt

Rate limit

Token bucket Redis (Lua) global entre todos os workers

Resiliência

Counters no Redis: HINCRBY atômico (O(1))

O progresso do job fica no hash job:{id}. Cada worker faz HINCRBY no servidor Redis, sem read-modify-write concorrente e sem entidade quente.

// RedisJobTracker — incremento atômico
await db.HashIncrementAsync($"job:{jobId}", field, 1);

// leitura: uma consulta ao hash
var values = await db.HashGetAllAsync($"job:{jobId}");
var completed = (sent + failed) >= total;
var status = completed ? "completed" : "processing";
O Redis também guarda o payload do comunicado e expira o hash em 24h. O worker usa IMemoryCache local por 5 min para reduzir leituras repetidas.
Resiliência

Idempotência: dedup nativa do Service Bus

Cada mensagem recebe um messageId determinístico: {jobId}:{md5hex(rowKey)}:{repeat}. A fila Standard tem requiresDuplicateDetection=true, então duplicatas dentro da janela são descartadas pelo broker.

  • messageId estável por job, usuário e repetição
  • Sem escrita extra por mensagem no caminho quente
  • Produtor usa EnqueueBatchAsync e ServiceBusMessageBatch
  • Consumidor usa PeekLock: Complete no sucesso/permanente; Abandon no transiente
// ServiceBusSendQueue.BuildMessage
return new ServiceBusMessage(json)
{
  ContentType = "application/json",
  MessageId = ServiceBusMessageId.Compute(
      body.JobId, body.RowKey, body.RepeatIndex)
};

// formato
"{jobId}:{md5hex(rowKey)}:{repeatIndex}"
DLQ nativa: a fila usa maxDeliveryCount=5 e lockDuration=PT5M; ao exceder o limite, o Service Bus move para $DeadLetterQueue.
Controle de vazão

Rate limit global: token bucket Redis (Lua)

O worker adquire 1 token no Redis antes de cada ContinueConversationAsync. Como todos os pods usam a mesma chave, o limite é global, não por réplica.

values.yamlDefaultSignificado
rateLimit.enabledtrueLiga/desliga o bucket
capacity50Burst máximo global
ratePerSec50Recarga por segundo
keyratelimit:botBucket compartilhado
 
-- TOKEN_BUCKET_LUA (resumo)
local capacity = tonumber(ARGV[1])
local rate = tonumber(ARGV[2])
local elapsed = math.max(0, now_ms - last) / 1000.0
local tokens = math.min(capacity, stored + elapsed * rate)
if tokens >= 1 then
  tokens = tokens - 1
  allowed = 1
end
redis.call('HSET', KEYS[1], 'tokens', tokens, 'ts', now_ms)
Resultado: o teto observado no teste 50k ficou no cap configurado (~52,5 msg/s), mesmo com KEDA escalando até 10 workers.
Segurança

Connection strings em Secret, TLS gerenciado, auth de tempo constante

Identidade & rede

  • Connection strings: Storage, Service Bus e Redis vêm do Secret do Kubernetes
  • TLS terminado no Istio Gateway com cert Let's Encrypt (auto-renew via cert-manager)
  • Secrets sensíveis em Secret do K8s — recomendado Key Vault + CSI em prod

API auth

  • /api/send, /api/jobs, /api/status exigem header x-api-key
  • Comparação com CryptographicOperations.FixedTimeEquals (timing-safe)
  • /api/messages protegido pelo token do próprio Bot Framework (JWT)
  • Dev local: Api:ApiKey vazia desliga a checagem (warning no log)
O único role assignment criado pela IaC é AcrPull para o kubelet no ACR compartilhado. O acesso data-plane aos serviços usa as connection strings do teams-msgs-secrets.

Fase 3 — Tratamento de erros

flowchart TD
  A["Worker recebe msg PeekLock"] --> B{"Acquire token Redis"}
  B --> C{"ContinueConversation"}
  C -->|200| D["HINCRBY sent + Complete"]
  C -->|429| E["Sleep Retry-After"] --> C
  C -->|5xx| F["Backoff exponencial"] --> C
  C -->|403/410| G["Remove ref + HINCRBY failed + Complete"]
  C -->|4xx outro| H["HINCRBY failed + Complete"]
  C -->|TaskCanceled/timeout| F
  C -->|transiente após retries| I["Abandon · broker reentrega"]
  I --> J{"DeliveryCount > max?"}
  J -->|não| A
  J -->|sim| K["DLQ nativa $DeadLetterQueue"]

Classificação de outcomes (SendWithRetry)

StatusAção
200IncrementSent + Complete
429Retry-After / backoff, até N
5xxBackoff exponencial, até N
403 / 410Remove ref + IncrementFailed (permanente)
4xx outroIncrementFailed (permanente)
transienteAbandon → reentrega pelo Service Bus
DLQ: após maxDeliveryCount=5, o broker move a mensagem para a subfila $DeadLetterQueue.
  • BotHttpStatus extrai o status real de ErrorResponseException.Response.StatusCode
  • TaskCanceledException de timeout é transiente; shutdown real ainda propaga
  • BotConnectorHttpClientFactory aplica timeout curto de 20s ao conector

Operação

Deploy em Azure — passo a passo

  1. App Registration do bot (az ad app create + sp + credential)
  2. Bicep: az deployment sub create → RG, Storage, Service Bus, Managed Redis, Log, AKS, Bot
  3. AcrPull: módulo acr-rbac.bicep no RG do ACR compartilhado
  4. AKS add-ons: Istio ingress externo + Gateway API CRDs v1.2.0
  5. cert-manager via Helm (jetstack) + DNS label no Public IP
  1. Aplica Gateway / HTTPRoute / ClusterIssuer / Certificate
  2. az acr build das imagens (api + worker) no ACR compartilhado
  3. values-poc.yaml: preenche image registry/tags, bot, x-api-key e as 3 connection strings
  4. Helm upgrade cria ConfigMap, Secret, Deployments e KEDA ScaledObject
  5. Atualiza o Messaging Endpoint do Azure Bot para o FQDN
helm upgrade --install teams-msgs deploy/helm/teams-msgs -n teams-msgs --create-namespace \
  -f deploy/helm/teams-msgs/values.yaml -f deploy/helm/teams-msgs/values-poc.yaml

Secret: Storage__ConnectionString, ServiceBus__ConnectionString e Redis__ConnectionString. Teams App: pwsh manifest/build.ps1 → upload no Admin Center → Setup policies (org-wide).

Resumo

Por que essa arquitetura importa

Escala com Redis

HINCRBY atômico elimina entidade quente; KEDA escala o worker 0→10 pela fila Service Bus.

Resiliência

Dedup nativa + DLQ do Service Bus, retry classificado e timeout curto no conector do Bot.

Segurança

Connection strings em Secret, TLS Let's Encrypt e x-api-key timing-safe.

IaC reproduzível

Bicep subscription-scope com 7 módulos; auth por connection string e AcrPull no ACR compartilhado.

Canal certo

Bot Framework proactive messaging — não burla rate limits, usa o canal nativo.

Testável

Helpers puros + 38 testes; load test de 50k validado no teto global de 50/s.
